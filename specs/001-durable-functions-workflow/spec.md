## Clarifications

### Session 2026-06-30

- Q: How should the orchestration instance ID be assigned — auto-generated, caller-supplied, or derived from the document identifier? → A: Derived deterministically from `documentId` (e.g., `"doc-{documentId}"`) to enable idempotent resubmission.
- Q: What URL route path should the HTTP trigger expose? → A: `POST /api/documents/process`.
- Q: What HTTP response should the trigger return when the request body cannot be deserialized or fails validation? → A: HTTP 400 with an RFC 9457 Problem Details body (`application/problem+json`, fields: `title`, `status`, `detail`).
- Q: Which Durable Functions storage backend should be used? → A: Azure Durable Task Scheduler (DTS) — the recommended backend replacing Azure Storage. Local development uses the DTS local emulator (Docker-based); cloud deployments use a provisioned DTS resource.
- Note: Clarifications Q1 (instance ID derived from `documentId`) and Q2 (route `POST /api/documents/process`) are superseded by `pipeline-flow.md`. The route is now `POST /api/process-documents` (container-scoped batch trigger); instance IDs are auto-generated by the runtime because the same container may be processed multiple times.
- Note: Azurite is required alongside the DTS emulator for local development — DTS replaces the Durable Functions task hub backend only; the `documents` Azure Storage Queue trigger still requires Azurite.
- Q: Which document types beyond invoices should the platform support? → A: Contracts (natural language clause extraction), BoundedDocument (single PDF containing multiple sub-documents — detect boundaries, process each segment), and General (schema-driven extraction from any format including PDF, Word, and scanned images). All three are supported as skeleton processing chains in this feature alongside the existing Invoice path.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Start a Document Batch Processing Run via HTTP (Priority: P1)

A developer or integration caller submits `POST /api/process-documents` with `{ "container_name": "documents" }`. The system returns HTTP 202 immediately with a processing handle. In the background, the root batch orchestration discovers all document folders in the blob container and processes all folders in parallel via one per-folder sub-orchestration each.

**Why this priority**: The HTTP trigger is the primary on-demand entry point. Without it, no document batch can be started and no downstream pipeline execution can be validated.

**Independent Test**: Post `{ "container_name": "documents" }` to the endpoint, receive HTTP 202, poll the returned status URL to `Completed`, and verify the output contains a `WorkflowResult` with folder-level sub-orchestration results.

**Acceptance Scenarios**:

1. **Given** the system is running locally, **When** `POST /api/process-documents` is called with `{ "container_name": "documents" }`, **Then** HTTP 202 is returned with a unique instance identifier and URLs for status querying, event signaling, termination, and history purge.
2. **Given** the orchestration completes, **When** the status endpoint is polled, **Then** the terminal status is `Completed` and the output contains a non-null `WorkflowResult` with aggregated sub-orchestration results.
3. **Given** the request body is missing `container_name` or cannot be deserialized, **When** the HTTP trigger receives it, **Then** HTTP 400 is returned with an RFC 9457 Problem Details body and no orchestration is started.

---

### User Story 2 - Root Batch Orchestration Fans Out Per-Folder Sub-Orchestrations (Priority: P2)

The root batch orchestration uses a folder discovery step to list all document-containing folders in the blob container, then starts one per-folder sub-orchestration per folder. All sub-orchestrations execute in parallel. When all complete, their results are aggregated into a single root `WorkflowResult`. In the skeleton phase, the folder discovery step returns a hardcoded placeholder list; no real blob operations occur.

**Why this priority**: The fan-out/fan-in pattern is the structural core of the pipeline. Establishing it correctly now ensures all future real step implementations can slot in without architectural refactoring.

**Independent Test**: Start an orchestration and verify that the root output `WorkflowResult` contains one entry per folder returned by the placeholder folder discovery step, confirming fan-out and fan-in occurred correctly.

**Acceptance Scenarios**:

1. **Given** the placeholder folder discovery step returns two folders, **When** the root orchestration processes the result, **Then** two per-folder sub-orchestrations are started and execute in parallel.
2. **Given** all sub-orchestrations complete, **When** the root orchestration collects results, **Then** all results are aggregated into a single `WorkflowResult` and the root orchestration completes with status `Completed`.
3. **Given** the folder discovery step returns an empty folder list, **When** the root orchestration evaluates the result, **Then** it starts zero sub-orchestrations and completes immediately with an empty `WorkflowResult`.

---

### User Story 3 - Per-Folder Sub-Orchestration Routes Each Document by Detected Type (Priority: P3)

Each per-folder sub-orchestration iterates over the document files in its assigned folder. For each document: (1) a classification step is called and returns a `ConfidenceResult<Classifications>`; (2) a result persistence step stores the classification result; (3) if `overallConfidence ≥ 0.8`, the appropriate type-specific processing chain runs by evaluating `PageClassifications` in fixed priority order — **Invoice → Contract → BoundedDocument → General → Email → None** — and executing only the chain for the first matching type found. Only one chain runs per document. Invoice documents are extracted and validated; Contract documents are extracted and validated; BoundedDocument documents trigger boundary detection followed by parallel per-segment sub-orchestrations; General documents are extracted. Documents with confidence below the threshold are logged and skipped with an error entry. Documents whose first matching type is `Email` or `None` are logged and skipped without an error entry. All steps are placeholder implementations with no real service calls.

**Note on mixed-content documents**: Detection of PDFs that contain multiple document types (e.g., an invoice followed by a contract) is the responsibility of the AI classifier, which is expected to return `BoundedDocument` when it detects page-boundary signatures within a single file. The routing logic does not attempt to detect or handle mixed content; it trusts the classifier's output.

**Why this priority**: The per-document type-dispatch pattern, conditional branching on confidence and classification type, and strongly typed step contracts form the core business logic skeleton. Establishing all branches now ensures real AI implementations can slot in without architectural refactoring across all supported document types.

**Independent Test**: Inspect the sub-orchestration execution history for placeholder documents of each type (Invoice, Contract, General); verify the correct processing chain steps appear in correct sequential order with strongly typed results for each type.

**Acceptance Scenarios**:

1. **Given** the classification step returns a placeholder result with `overallConfidence ≥ 0.8` and `classification = "Invoice"`, **When** the sub-orchestration evaluates it, **Then** a result persistence step, the invoice extraction step, a result persistence step, the invoice validation step, and a final result persistence step are called in that order for that document.
2. **Given** the classification step returns a placeholder result with `overallConfidence ≥ 0.8` and `classification = "Contract"`, **When** the sub-orchestration evaluates it, **Then** a result persistence step, the contract extraction step, a result persistence step, the contract validation step, and a final result persistence step are called in that order for that document.
3. **Given** the classification step returns a placeholder result with `overallConfidence ≥ 0.8` and `classification = "General"`, **When** the sub-orchestration evaluates it, **Then** a result persistence step, the general document extraction step, and a final result persistence step are called in that order for that document.
4. **Given** the classification step returns a placeholder result with `overallConfidence < 0.8`, **When** the sub-orchestration evaluates it, **Then** further processing for that document is skipped and an error entry is added to the sub-orchestration's `WorkflowResult`.
5. **Given** the classification step returns a classification of `Email` or `None`, **When** the sub-orchestration evaluates it, **Then** that document is skipped with a log message and the sub-orchestration continues to the next document.

### User Story 4 - Storage Queue Trigger Starts Batch Processing (Priority: P4)

A Base64-encoded `{ "container_name": "documents" }` message placed on the `documents` storage queue triggers the same root batch orchestration as the HTTP trigger, enabling automated batch scheduling without requiring a direct HTTP caller.

**Why this priority**: The queue trigger is the production ingestion path for scheduled or event-driven runs. Including it in the skeleton ensures the trigger-to-orchestration wiring is validated before real document processing is introduced.

**Independent Test**: Publish a Base64-encoded `{ "container_name": "documents" }` message to the local `documents` queue and verify that a new root batch orchestration starts and reaches `Completed` status.

**Acceptance Scenarios**:

1. **Given** the system is running locally, **When** a valid Base64-encoded `{ "container_name": "documents" }` message is placed on the `documents` queue, **Then** a new root batch orchestration starts and reaches `Completed` status.
2. **Given** a malformed or undecodable queue message is received, **When** the trigger processes it, **Then** the message is dead-lettered after retry exhaustion; no orchestration is started.

---

### User Story 5 - Bounded/Continuous Document Boundary Detection and Per-Segment Processing (Priority: P5)

When a document is classified as `BoundedDocument` with sufficient confidence, the per-folder sub-orchestration runs a boundary detection step that returns a `BoundaryDetectionResult` containing a list of `DocumentSegment` entries. One per-segment sub-orchestration is started per detected segment and all execute in parallel. Each per-segment sub-orchestration runs the same type-specific processing chain (classification → extraction → validation → persistence) for its assigned page range. When all segment sub-orchestrations complete, their results are aggregated into the parent sub-orchestration's `WorkflowResult`. In the skeleton phase, the boundary detection step returns a hardcoded list of two segments and no real document splitting occurs.

**Why this priority**: Bounded/continuous document support introduces a nested fan-out pattern — a sub-orchestration spawning further sub-orchestrations. Establishing this structural skeleton in the foundation phase ensures the pattern is available for real document splitting implementations without architectural rework.

**Independent Test**: Start an orchestration where a placeholder document is classified as `BoundedDocument`; verify that the boundary detection step is called, two per-segment sub-orchestrations are started in parallel, both reach `Completed` status, and their results are aggregated into the parent sub-orchestration's `WorkflowResult`.

**Acceptance Scenarios**:

1. **Given** the classification step returns a placeholder result with `overallConfidence ≥ 0.8` and `classification = "BoundedDocument"`, **When** the sub-orchestration evaluates it, **Then** the boundary detection step is called and one per-segment sub-orchestration is started per detected segment.
2. **Given** the placeholder boundary detection step returns two segments, **When** the per-folder sub-orchestration processes the result, **Then** two per-segment sub-orchestrations execute in parallel and both reach `Completed` status.
3. **Given** the boundary detection step returns zero segments, **When** the per-folder sub-orchestration evaluates the result, **Then** no per-segment sub-orchestrations are started and an error entry is added to the `WorkflowResult` for that document.

### Edge Cases

- What happens when the HTTP trigger receives a request body that cannot be deserialized? → HTTP 400 is returned with an RFC 9457 Problem Details body; no orchestration is started.
- What happens if the folder discovery step returns an empty folder list? → The root batch orchestration starts zero sub-orchestrations and completes immediately with an empty `WorkflowResult`.
- What happens if a placeholder processing step throws an unhandled exception? → The orchestration engine retries per the configured retry policy; after retry exhaustion the orchestration transitions to `Failed` status; the failure is logged with orchestration instance ID and correlation ID.
- What happens if one sub-orchestration fails while others are still running? → All sub-orchestrations run to completion or failure in parallel; failed sub-orchestrations contribute error entries to the aggregated `WorkflowResult`; the root orchestration can still complete with partial results.
- What happens when boundary detection returns zero segments for a BoundedDocument? → No per-segment sub-orchestrations are started; an error entry is added to the sub-orchestration's `WorkflowResult` and processing continues to the next document.
- What happens when a segment within a BoundedDocument is itself classified as `BoundedDocument`? → Nested BoundedDocument classification is not supported in the skeleton phase; the segment is processed using the General document path as a fallback.
- How does the status query endpoint respond when an unknown instance ID is queried? → HTTP 404 per the orchestration management API contract.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST expose an HTTP POST endpoint at route `POST /api/process-documents` that accepts `{ "container_name": "<name>" }` in the request body, starts a root batch orchestration, and returns HTTP 202 with a unique instance identifier and URLs for status querying, event signaling, termination, and history purge.
- **FR-001a**: The system MUST expose a storage queue trigger on the `documents` queue that accepts a Base64-encoded JSON message with `{ "container_name": "<name>" }` and starts the same root batch orchestration as the HTTP trigger.
- **FR-002**: The batch orchestration instance identifier MUST be auto-generated by the system; callers MUST NOT supply or predict it, because the same container may be submitted for processing multiple times.
- **FR-003**: The root batch orchestration MUST: (1) retrieve the list of document folders from the configured container via a folder discovery step; (2) start one per-folder sub-orchestration per folder, with all sub-orchestrations executing in parallel; (3) collect all sub-orchestration results and aggregate them into a single root `WorkflowResult`.
- **FR-004**: Each per-folder sub-orchestration MUST process each document file in its folder sequentially: (1) run the classification step and obtain a `ConfidenceResult<Classifications>`; (2) run a result persistence step to store the classification result; (3) if `overallConfidence < threshold`: log the skip reason, add an error entry to the sub-orchestration's `WorkflowResult`, and continue to the next file; (4) otherwise, evaluate `PageClassifications` in fixed priority order — **Invoice → Contract → BoundedDocument → General → Email → None** — and execute only the chain for the **first matching type** found; exactly one chain runs per document:
  - First match is `"Invoice"`: run the invoice extraction step, persist the extraction result, run the invoice validation step, and persist the validation result.
  - First match is `"Contract"`: run the contract extraction step, persist the extraction result, run the contract validation step, and persist the validation result.
  - First match is `"BoundedDocument"`: run the boundary detection step to obtain a `BoundaryDetectionResult`; for each detected `DocumentSegment`, start one per-segment sub-orchestration in parallel; wait for all segment sub-orchestrations to complete; aggregate and persist the combined result. If zero segments are detected, add an error entry to the `WorkflowResult` without starting any sub-orchestrations.
  - First match is `"General"`: run the general document extraction step and persist the extraction result.
  - First match is `"Email"` or `"None"`: log the skip reason and continue to the next file without adding an error entry.

  **Classifier responsibility**: Detection of mixed-content PDFs (a single file containing multiple document types) is the responsibility of the AI classifier, which MUST return `"BoundedDocument"` when it detects page-boundary signatures within a file. The routing logic trusts the classifier's output and does not inspect all page classifications for multiple matching types.
- **FR-005**: In the skeleton phase, ALL processing steps MUST be placeholder implementations: the folder discovery step returns two hardcoded folders each containing one document path; the classification step returns a hardcoded result with confidence 1.0 and one Invoice page classification; the invoice extraction step returns a hardcoded `ConfidenceResult<Invoice>` with confidence 1.0; the invoice validation step returns a hardcoded valid `ValidationResult`; the contract extraction step returns a hardcoded `ConfidenceResult<ContractData>` with confidence 1.0; the contract validation step returns a hardcoded valid `ValidationResult`; the boundary detection step returns a hardcoded `BoundaryDetectionResult` with two segments covering distinct page ranges; the general document extraction step returns a hardcoded `ConfidenceResult<GeneralDocumentData>` with confidence 1.0 and two sample fields; the result persistence step returns success without performing any I/O.
- **FR-006**: The confidence threshold MUST be defined as a named constant with value 0.8 and referenced symbolically throughout both orchestrations; it MUST NOT appear as an inline literal.
- **FR-007**: Orchestration coordination code MUST be side-effect free and produce the same outcome on any re-execution given the same recorded state. System time reads, unique identifier generation, I/O operations, and any other non-deterministic call MUST be delegated to discrete processing steps; they MUST NOT occur inline within orchestration coordination logic.
- **FR-008**: The orchestration instance identifier and a correlation identifier MUST be included as structured log properties in every log entry emitted by the HTTP trigger, queue trigger, both orchestrations, and all processing steps.
- **FR-009**: If no correlation identifier is present in the incoming HTTP request headers (`X-Correlation-Id`), the system MUST generate one automatically and propagate it throughout the entire orchestration.
- **FR-010**: The solution MUST be executable locally without dependency on any deployed cloud resources, using local emulators for durable orchestration state and for queue-based and blob storage.
- **FR-011**: Orchestration classes and processing step implementations MUST be organized into the architectural layers defined in the Engineering Constitution. Trigger code MUST contain no business logic.
- **FR-012**: The solution MUST conform to the technology stack mandated by the Engineering Constitution. Technology choices are fixed by the Constitution and are not subject to re-evaluation in this spec.
- **FR-013**: Unit tests MUST be provided covering: root orchestration fan-out with two folders and with zero folders; per-folder sub-orchestration full step chain for an Invoice document; per-folder sub-orchestration full step chain for a Contract document; per-folder sub-orchestration full step chain for a General document; per-folder sub-orchestration boundary detection and segment fan-out for a BoundedDocument with two segments; per-folder sub-orchestration zero-segment handling for a BoundedDocument; per-folder sub-orchestration skip logic for confidence below threshold; per-folder sub-orchestration skip logic for Email and None classifications; and each placeholder processing step's return contract.
- **FR-014**: When the HTTP trigger receives a request body that cannot be deserialized or is missing `container_name`, it MUST return HTTP 400 conforming to RFC 9457 Problem Details (`Content-Type: application/problem+json`) with `title`, `status` (400), and `detail`. No orchestration must be started for invalid requests.

### Key Entities

- **DocumentBatchRequest**: Inbound trigger payload — contains `containerName` (string). Used by both the HTTP trigger and the queue trigger.
- **DocumentFolder**: A folder entry within the blob container — contains `containerName` (string), `name` (string), `documentFileNames` (list of strings).
- **DocumentFolders**: Output of the folder discovery step — contains `folders` (list of `DocumentFolder`).
- **ConfidenceResult\<T\>**: Generic result wrapper — contains `data` (of type T) and `overallConfidence` (numeric, 0.0–1.0). Confidence is derived from AI model output signals in production; hardcoded in the skeleton phase.
- **PageClassification**: A single page range classification — contains `classification` (string: `"Invoice"` | `"Contract"` | `"BoundedDocument"` | `"General"` | `"Email"` | `"None"`), `imageRangeStart` (optional integer), `imageRangeEnd` (optional integer).
- **Classifications**: Output of the classification step — contains `pageClassifications` (list of `PageClassification`).
- **Address**: Postal address — contains `street`, `city`, `state`, `postalCode`, `country` (all strings).
- **MonetaryAmount**: A monetary value — contains `amount` (numeric) and `currency` (string).
- **InvoiceItem**: A single line item on an invoice — contains `productCode` (string), `description` (string), `quantity` (numeric), `unitPrice` (`MonetaryAmount`), `total` (`MonetaryAmount`), `tax` (optional `MonetaryAmount`).
- **Invoice**: Extracted invoice data — contains `invoiceId`, `customerName`, `vendorName`, `invoiceDate`, `dueDate` (string, YYYY-MM-DD format), `customerAddress` (`Address`), `invoiceTotal` (`MonetaryAmount`), `items` (list of `InvoiceItem`); optional fields: `purchaseOrder`, `subtotal`, `totalTax`, `paymentTerm`.
- **ValidationResult**: Output of an invoice or contract validation step — contains `isValid` (boolean), `messages` (list of strings), and nullable per-rule status flags whose presence depends on the validator: `hasInvoiceId` (boolean?) — set by invoice validation, true when the extracted invoice contains a non-empty `invoiceId`; `allItemsValid` (boolean?) — set by invoice validation, true when every line item passes cost and quantity checks. Additional validator-specific flags will be added alongside each real validator implementation. A null flag value means the rule was not evaluated by the current validator.
- **ContractData**: Extracted contract data — contains `parties` (list of strings), `effectiveDate` (string, YYYY-MM-DD format), `expirationDate` (optional string, YYYY-MM-DD format), `renewalTerms` (optional string), `exitClause` (optional string), `governingLaw` (optional string), `keyObligations` (list of strings).
- **DocumentSegment**: A detected sub-document within a bounded/continuous document — contains `segmentIndex` (integer), `pageStart` (integer), `pageEnd` (integer), `detectedType` (string, one of the classification values).
- **BoundaryDetectionResult**: Output of the boundary detection step for bounded/continuous documents — contains `segments` (list of `DocumentSegment`).
- **GeneralDocumentData**: Structured data extracted from a general document — contains `schemaName` (string) and `fields` (list of name-value pairs where each entry has `name` (string) and `value` (string)).
- **WorkflowResult**: Aggregated output of a per-folder sub-orchestration or the root batch orchestration — contains `messages` (list of strings) and `errors` (list of strings).

## Azure Services Involved *(mandatory for cloud features)*

- **Primary Services**: Azure Functions (Isolated Worker, .NET 10), Azure Durable Functions, Azure Durable Task Scheduler (DTS), Azure Blob Storage (documents container), Azure Storage Queue (`documents` queue)
- **Supporting Services**: Azure OpenAI GPT-4o (multimodal — placeholder in skeleton), Azure AI Document Intelligence prebuilt-layout (OCR — placeholder in skeleton), Application Insights (telemetry sink), Azure Monitor (log aggregation)
- **Service Responsibilities**:
  - Azure Functions: Hosts the HTTP trigger, queue trigger, both orchestrations (root batch and per-folder), and all processing step implementations
  - Azure Durable Functions: Provides stateful orchestration, sub-orchestration, parallel fan-out, history persistence, and the orchestration management HTTP API
  - Azure Durable Task Scheduler (DTS): Provides the durable orchestration state backend. The DTS local emulator is used for local development.
  - Azure Blob Storage: Stores source documents (read by the folder discovery and classification steps in production) and receives result JSON blobs written by the result persistence step. Placeholder steps simulate these operations in the skeleton phase; Azurite is used locally.
  - Azure Storage Queue (`documents`): Receives trigger messages for scheduled or automated batch processing. Azurite is used locally.
  - Azure OpenAI (GPT-4o): Used by the classification step (vision-based page classification with confidence signals) and the invoice extraction step (structured multimodal extraction) in production. Placeholder steps return hardcoded typed results in the skeleton.
  - Azure AI Document Intelligence (prebuilt-layout): Used by the invoice extraction step for OCR text extraction and per-word confidence signals in production. Placeholder steps return hardcoded results in the skeleton.
  - Application Insights: Receives structured telemetry for operational monitoring
- **Regional/Residency Constraints**: No data residency constraints for the skeleton phase; placeholder activities process no real document data.

## Security Considerations *(mandatory)*

- **Data Classification**: Internal — source PDF documents may contain business-sensitive invoice data; result blobs contain extracted invoice fields including financial figures. In the skeleton phase only synthetic placeholder data is processed.
- **Data Protection**: Source PDFs and extracted invoice data in transit must use TLS. Data at rest in Azure Blob Storage must use Azure-managed encryption. In the skeleton phase no real documents are accessed.
  - **Secret Handling**: Local emulators require no secrets. Cloud environment configuration must store all service connection strings and keys (Azure Blob Storage, Azure OpenAI endpoint and key, Azure AI Document Intelligence endpoint and key, DTS connection string) in Azure Key Vault and reference them via App Settings; secrets must never appear in source control. A local configuration template file will be committed; the actual local configuration file must be git-ignored.
- **Threat Considerations**: The HTTP trigger endpoint is unauthenticated in the local development profile. Cloud deployment must require authentication (Azure Functions host key or Azure AD) before production use; authentication is architecturally deferred to a subsequent phase but must not be structurally prevented. Extracted invoice data is sensitive financial information; access to result blobs must be restricted to authorized principals only.
- **Compliance Needs**: Invoice data may be subject to financial recordkeeping requirements. Compliance will be addressed when the platform processes real documents.

## Identity and RBAC *(mandatory)*

- **Workload Identity**: The Functions App will use a system-assigned Managed Identity for cloud deployments. Local emulators require no identity.
- **Human Access Model**: Developers access the local Functions host directly via HTTP and via Azurite queues on localhost. No Azure portal access is required for local execution.
- **RBAC Roles Required**:
  - Functions App Managed Identity: `Durable Task Data Contributor` on the Durable Task Scheduler resource — required for Durable Functions task hub read/write operations
  - Functions App Managed Identity: `Storage Blob Data Reader` on the source documents blob container — required for the folder discovery and classification steps to read documents in production
  - Functions App Managed Identity: `Storage Blob Data Contributor` on the results blob container — required for the result persistence step to write result JSON blobs in production
  - Functions App Managed Identity: `Storage Queue Data Consumer` on the Storage Account hosting the `documents` queue — required for the queue trigger
  - Functions App Managed Identity: `Cognitive Services OpenAI User` on the Azure OpenAI resource — required for the classification and invoice extraction steps in production
  - Functions App Managed Identity: `Cognitive Services User` on the Azure AI Document Intelligence resource — required for the invoice extraction step in production
- **Least Privilege Plan**: Each role is scoped to the minimum required resource and access level. No subscription-level roles are assigned. All role assignments will be provisioned via Bicep in a follow-on infrastructure feature.

## Observability Requirements *(mandatory)*

- **Structured Logging**: Every log entry MUST include `orchestrationInstanceId` and `correlationId` as structured properties. Key events: orchestration started, each activity started and completed, orchestration completed or failed.
- **Distributed Tracing**: An OpenTelemetry activity span should exist for the HTTP trigger invocation and for each Durable Functions activity execution, correlated by orchestration instance ID.
- **Metrics**: Activity execution duration and overall orchestration end-to-end duration must be observable via Application Insights telemetry in cloud deployments.
- **Dashboards/Workbooks**: No new dashboards required for the skeleton phase. Telemetry must be structured to enable future dashboards without re-instrumentation.
- **Alerting**: No automated alerts required for the skeleton phase. Alerting will be introduced when the platform processes real documents.

## Performance Expectations *(mandatory)*

- **Latency Targets**: The HTTP trigger MUST return HTTP 202 in under 2 seconds for a valid request under no load on a developer machine. The complete root orchestration (all placeholder steps across two folders) MUST complete in under 15 seconds using local emulators.
- **Throughput Targets**: No throughput requirements for the skeleton phase. Throughput targets will be defined when real step implementations are introduced.
- **Scalability Expectations**: The orchestration design MUST NOT include assumptions that prevent horizontal scaling of the host in future deployments.
- **Resilience Targets**: Processing steps MUST be idempotent by design (placeholder implementations trivially satisfy this). Retry policy configuration points must be present in the orchestration code even if set to defaults for this phase.

## Cost Considerations *(mandatory)*

- **Cost Drivers**: Azure Functions execution time (per-execution billing), Durable Functions task hub operations against the Azure Durable Task Scheduler, Azure Blob Storage read/write operations (minimal — placeholder activities perform no real I/O in the skeleton phase), Azure OpenAI token usage (none in skeleton phase), Azure AI Document Intelligence page processing (none in skeleton phase).
- **Estimated Cost Impact**: Low. Placeholder activities perform no real computation or I/O; execution time per orchestration is negligible.
- **Cost Controls**: No special cost controls needed at the skeleton phase. Concurrency limits and per-document cost budgets will be applied when real workloads are introduced.
- **Cost Monitoring Plan**: Application Insights telemetry captures execution duration per activity, enabling cost attribution per document processing run in future phases.

## Risks *(mandatory)*

- **Risk**: Orchestration coordination code violates the side-effect-free rule (e.g., system time, unique ID generation, or I/O performed directly in orchestration logic rather than delegated to processing steps).
  - **Likelihood**: Medium
  - **Impact**: High — non-deterministic orchestrations can replay incorrectly, producing data corruption or infinite replay loops
  - **Mitigation**: Code review checklist item for every orchestration PR; the determinism constraint is documented in this spec and in the Engineering Constitution; static analysis tooling should flag non-deterministic patterns in orchestration code
  - **Contingency**: Orchestration history can be purged and the orchestration restarted if replay corruption is detected early

- **Risk**: Processing step result contracts are changed incompatibly after downstream callers have taken a dependency.
  - **Likelihood**: Low
  - **Impact**: Medium — contract breakage requires coordinated updates across orchestration and callers
  - **Mitigation**: Result types are defined in the shared contracts layer; changes require a contract versioning review per the contract versioning policy
  - **Contingency**: Introduce a versioned result type and maintain the original type for a compatibility window

- **Risk**: The DTS local emulator behaviour diverges from the cloud DTS service, masking failures that only surface in deployed environments.
  - **Likelihood**: Low — the DTS emulator is maintained by Microsoft to be compatible with the cloud service
  - **Impact**: Medium — integration failures discovered late in the delivery cycle
  - **Mitigation**: The local configuration template is maintained in sync with cloud configuration shapes; integration tests should target both the local emulator and a cloud DTS resource in CI
  - **Contingency**: A dedicated integration test environment with a real DTS resource can be used to reproduce and diagnose cloud-only failures

## Assumptions

- The existing solution file will be used as the container for all projects; no new solution file is needed.
- The Functions host project (`DocumentProcessing.Functions`) will be configured as the entry point for triggers and will contain no business logic.
- The Application project (`DocumentProcessing.Application`) will host both orchestrations and orchestration coordination logic; the Core project (`DocumentProcessing.Core`) will host domain entities.
- The Contracts project (`DocumentProcessing.Contracts`) will host shared types (data transfer objects and processing step contracts) used across project boundaries.
- Local development uses local emulators for durable orchestration state and for queue and blob storage; no deployed cloud resources are required.
- A local configuration template file will be committed; the actual local configuration file must be git-ignored.
- The blob container name `documents` and queue name `documents` are the default values for local development; both are configurable via App Settings.
- The task hub name is configurable via App Settings and will use the same value across all environments.
- The placeholder folder discovery step will return two hardcoded folders each containing one document path, enabling fan-out and sub-orchestration testing without real blob storage.
- The placeholder classification step returns maximum confidence and an Invoice classification by default, exercising the Invoice processing chain end-to-end. Unit tests inject other classification types (Contract, BoundedDocument, General) directly to exercise all processing branches without requiring separate end-to-end runs for each type.
- The placeholder boundary detection step returns two hardcoded segments with distinct page ranges, enabling per-segment fan-out to be exercised without real document splitting.
- The placeholder contract extraction step returns a hardcoded `ContractData` result with sample parties, effective date, and renewal terms.
- The placeholder general document extraction step returns a hardcoded `GeneralDocumentData` result with a sample schema name and two sample name-value field pairs.
- Cloud infrastructure Bicep is out of scope for this feature; only local execution is required to be functional at completion.
