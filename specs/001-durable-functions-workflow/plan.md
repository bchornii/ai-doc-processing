# Implementation Plan: Durable Functions Workflow Foundation

**Branch**: `001-durable-functions-workflow` | **Date**: 2026-07-01 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-durable-functions-workflow/spec.md`

## Summary

Implement the complete Durable Functions workflow skeleton for the AI Document Processing platform. The implementation establishes HTTP and Queue triggers, a root batch orchestration with fan-out/fan-in across document folders, per-folder sub-orchestrations with first-match priority-order type routing (Invoice → Contract → BoundedDocument → General → Email → None), and nested sub-orchestrations for segmented documents. All processing steps are placeholder implementations returning hardcoded typed results. The orchestration backend is Azure Durable Task Scheduler (DTS), using the local emulator for development. See [research.md](research.md) for all resolved technical decisions.

## Durable Functions Orchestration

**Orchestration Pattern**: Async HTTP API + Fan-Out/Fan-In + Sub-orchestrations (nested)

**Orchestrators**:

| Orchestrator | Trigger | Input | Output | Completion Criteria |
|---|---|---|---|---|
| `ProcessDocumentBatchWorkflow` | HTTP trigger or Queue trigger | `ProcessDocumentBatchInput` (`DocumentBatchRequest` + `correlationId`) | `WorkflowResult` (aggregated) | All per-folder sub-orchestrations complete |
| `ProcessDocumentWorkflow` | Sub-orchestration start by root | `ProcessDocumentFolderInput` (`DocumentFolder` + `correlationId`) | `WorkflowResult` | All documents in folder processed |
| `ProcessDocumentSegmentWorkflow` | Sub-orchestration start by per-folder orchestrator | `DocumentSegmentInput` (`DocumentSegment` + `ContainerName` + `FolderName` + `BlobName` + `correlationId`) | `WorkflowResult` | Segment classified, extracted, validated, persisted |

**Activity Functions**:

| Activity | Input | Output | Idempotency | Retry Policy |
|---|---|---|---|---|
| `GetDocumentFolders` | `ProcessDocumentBatchInput` | `DocumentFolders` | Idempotent (read-only; hardcoded in skeleton) | `TaskOptions` retry (see below) |
| `ClassifyDocument` | Container name, blob name, correlationId | `ConfidenceResult<Classifications>` | Idempotent (hardcoded in skeleton) | `TaskOptions` retry |
| `PersistResult` | Container, blob name, content (`byte[]` UTF-8 JSON), correlationId | `bool` (success) | Idempotent (blob upsert; no-op in skeleton) | `TaskOptions` retry |
| `ExtractInvoice` | Container name, blob name, page range, correlationId | `ConfidenceResult<Invoice>` | Idempotent (hardcoded in skeleton) | `TaskOptions` retry |
| `ValidateInvoice` | Invoice name, `Invoice` data, correlationId | `ValidationResult` | Idempotent (pure logic) | `TaskOptions` retry |
| `ExtractContract` | Container name, blob name, page range, correlationId | `ConfidenceResult<ContractData>` | Idempotent (hardcoded in skeleton) | `TaskOptions` retry |
| `ValidateContract` | Contract name, `ContractData`, correlationId | `ValidationResult` | Idempotent (pure logic) | `TaskOptions` retry |
| `DetectBoundaries` | Container name, blob name, correlationId | `BoundaryDetectionResult` | Idempotent (hardcoded in skeleton) | `TaskOptions` retry |
| `ExtractGeneralDocument` | Container name, blob name, page range, correlationId | `ConfidenceResult<GeneralDocumentData>` | Idempotent (hardcoded in skeleton) | `TaskOptions` retry |

> **Retry policy location**: `TaskOptions(new RetryPolicy(maxNumberOfAttempts: 3, firstRetryInterval: TimeSpan.FromSeconds(5), backoffCoefficient: 2.0))` passed to every `CallActivityAsync` in orchestrator code (T040, T048, T050). `host.json` is **not** used for Durable activity retries.

**Root Orchestrator Flow** (`ProcessDocumentBatchWorkflow`):
```
1. CallActivityAsync<DocumentFolders>(GetDocumentFolders, input, taskOptions)
2. if Folders is empty => return WorkflowResult.Empty
3. Fan-out: foreach folder => CallSubOrchestrationAsync<WorkflowResult>(
       ProcessDocumentWorkflow,
       new ProcessDocumentFolderInput(folder, input.CorrelationId))
4. Fan-in: await Task.WhenAll => collect all WorkflowResult
5. Aggregate all results => return root WorkflowResult
```

**Per-Folder Orchestrator Flow** (`ProcessDocumentWorkflow`):
```
foreach documentFileName in input.Folder.DocumentFileNames:
  1. result = CallActivityAsync<ConfidenceResult<Classifications>>(
         ClassifyDocument, {folder.ContainerName, blobName, correlationId}, taskOptions)
  2. CallActivityAsync<bool>(PersistResult,
         {classification result as UTF-8 byte[]}, taskOptions)
  3. if result is null => log error, add error entry, continue
  4. if result.OverallConfidence < ConfidenceThreshold => log skip, add error entry, continue
  5. Evaluate PageClassifications in fixed priority order — first match wins (FR-004):
     Invoice    => ExtractInvoice => PersistResult => ValidateInvoice => PersistResult
     Contract   => ExtractContract => PersistResult => ValidateContract => PersistResult
     BoundedDoc => DetectBoundaries;
                   if Segments.Count == 0 => add error entry, continue
                   else fan-out CallSubOrchestrationAsync(ProcessDocumentSegmentWorkflow,
                        new DocumentSegmentInput(segment, containerName, folderName,
                                                 blobName, correlationId)) per segment
                        + Task.WhenAll fan-in + PersistResult
     General    => ExtractGeneralDocument => PersistResult
     Email/None => log, continue (no error entry)
return WorkflowResult (aggregated messages + errors)
```

**Mixed-content PDF note**: Detection of multi-type documents (e.g., invoice + contract in one PDF) is the classifier's responsibility — it returns `BoundedDocument` when it detects page-boundary signatures. The routing logic does not inspect multiple page types per document; exactly one chain runs per document.

**Per-Segment Orchestrator Flow** (`ProcessDocumentSegmentWorkflow`):
```
1. ClassifyDocument (input.BlobName + input.Segment page range, taskOptions)
2. PersistResult (classification as UTF-8 byte[])
3. if OverallConfidence < ConfidenceThreshold => add error, return
4. Evaluate PageClassifications in same priority order (first match wins):
   Invoice    => ExtractInvoice => PersistResult => ValidateInvoice => PersistResult
   Contract   => ExtractContract => PersistResult => ValidateContract => PersistResult
   BoundedDoc => fallback to General path (nested BoundedDocument unsupported per spec)
   General    => ExtractGeneralDocument => PersistResult
   Email/None => log, skip
5. return WorkflowResult
```

**State and Determinism Considerations**:
- Orchestrators MUST NOT call `DateTime.UtcNow`, `Guid.NewGuid()`, `Random`, or any I/O
- Use `context.CreateReplaySafeLogger()` for all logging within orchestrators
- Correlation ID is passed as part of the orchestration input (set by trigger); not read mid-replay
- All non-deterministic operations are in activities

**Error Handling and Compensation**:
- Transient activity failures: `TaskOptions(new RetryPolicy(3, TimeSpan.FromSeconds(5), backoffCoefficient: 2.0))` per `CallActivityAsync` in orchestrator code
- Classification null result: log error, add error entry, continue to next document
- Confidence below threshold: log, add error entry, continue to next document
- Empty segment list from `DetectBoundaries`: add error entry, continue (per FR-004)
- Queue trigger invalid/undecodable message: log error, rethrow — triggers dead-lettering after retry exhaustion

## Security

**Identity and Access**:
- Local development: emulators require no identity
- Cloud (follow-on feature): system-assigned Managed Identity on the Functions App
- RBAC roles required in cloud: `Durable Task Data Contributor` (DTS), `Storage Blob Data Reader` (source docs container), `Storage Blob Data Contributor` (results container), `Storage Queue Data Consumer` (documents queue), `Cognitive Services OpenAI User` (Azure OpenAI), `Cognitive Services User` (Document Intelligence)
- All role assignments provisioned via Bicep in the follow-on infrastructure feature

**Secret and Key Management**:
- `local.settings.json` git-ignored; `local.settings.template.json` committed (FR-010)
- Cloud: all connection strings and keys in Azure Key Vault, referenced via App Settings
- No secrets in source control (Constitution V)

**Network and Data Protection**:
- Skeleton phase: no real documents processed; no data residency constraints
- Cloud: TLS for all external communication; Azure-managed encryption at rest for Blob Storage
- HTTP trigger: unauthenticated in local profile; cloud authentication (host key or Azure AD) architecturally deferred but structurally unblocked

**Threat Considerations**:
- Extracted invoice/contract data is sensitive financial information; result blob access restricted to authorized principals in cloud deployments
- Queue trigger: malformed messages dead-lettered after retry exhaustion; no orchestration started for invalid input

## Monitoring

**Structured Logging**: Every log entry in triggers, orchestrations, and activity wrappers MUST include `orchestrationInstanceId` and `correlationId` as structured log properties (FR-008). Use `ILogger.BeginScope` or named parameters. Use `context.CreateReplaySafeLogger()` in all orchestrators (FR-007).

**Application Insights**: `Microsoft.Azure.Functions.Worker.ApplicationInsights` provides auto-instrumentation for activity dependency tracking and host-level metrics. No manual span or metric code required in the skeleton phase — Application Insights auto-collects execution telemetry, and the DTS dashboard provides orchestration-level activity timing.

**Distributed Tracing**: Correlation ID propagated through the orchestration input chain to all activity calls, enabling cross-component log correlation in Application Insights queries.

**Dashboards/Alerting**: No new dashboards or alerts in the skeleton phase. DTS dashboard provides orchestration health and activity timing. Telemetry structured for future Application Insights workbooks without re-instrumentation.

## Testing Strategy

**Unit Tests** (FR-013, T051–T061, T064):
- Root orchestration fan-out (two folders, zero folders)
- Per-folder sub-orchestration: full chain for each type (Invoice, Contract, General, BoundedDocument with two segments, BoundedDocument with zero segments)
- Per-folder sub-orchestration skip logic (confidence below threshold, Email, None)
- Each placeholder activity return contract
- HTTP trigger: valid request → 202, invalid request → 400 Problem Details
- Queue trigger: valid message → orchestration started, malformed message → no orchestration + rethrow

**Integration Tests** (T065):
- Local smoke test: start Functions host in-process, POST to `/api/process-documents`, poll `statusQueryGetUri` to `Completed`, assert `WorkflowResult` is non-null with zero errors
- Requires DTS local emulator and Azurite running

**Quality Gates**:
- `dotnet build` succeeds with no errors (Phase 1 gate)
- `dotnet test` exits 0 with all unit tests passing (Phase 8 gate)
- Integration smoke test passes against local emulators
