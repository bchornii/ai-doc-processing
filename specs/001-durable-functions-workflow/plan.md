# Implementation Plan: Durable Functions Workflow Foundation

**Branch**: `001-durable-functions-workflow` | **Date**: 2026-06-30 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-durable-functions-workflow/spec.md`

## Summary

Implement the complete Durable Functions workflow skeleton for the AI Document Processing platform. The implementation establishes HTTP and Queue triggers, a root batch orchestration with fan-out/fan-in across document folders, per-folder sub-orchestrations with type-dispatch routing (Invoice, Contract, BoundedDocument, General), and nested sub-orchestrations for segmented documents. All processing steps are placeholder implementations returning hardcoded typed results. The orchestration backend is Azure Durable Task Scheduler (DTS), using the local emulator for development. See [research.md](research.md) for all resolved technical decisions.

## Durable Functions Orchestration

**Orchestration Pattern**: Async HTTP API + Fan-Out/Fan-In + Sub-orchestrations (nested)

**Orchestrators**:

| Orchestrator | Trigger | Input | Output | Completion Criteria |
|---|---|---|---|---|
| `ProcessDocumentBatchWorkflow` | HTTP trigger or Queue trigger | `DocumentBatchRequest` + `correlationId` | `WorkflowResult` (aggregated) | All per-folder sub-orchestrations complete |
| `ProcessDocumentWorkflow` | Sub-orchestration start by root | `DocumentFolder` + `correlationId` | `WorkflowResult` | All documents in folder processed |
| `ProcessDocumentSegmentWorkflow` | Sub-orchestration start by per-folder orchestrator | `DocumentSegment` + `DocumentFolder` context + `correlationId` | `WorkflowResult` | Segment classified, extracted, validated, persisted |

**Activity Functions**:

| Activity | Input | Output | Idempotency | Retry Policy |
|---|---|---|---|---|
| `GetDocumentFolders` | `DocumentBatchRequest` | `DocumentFolders` | Idempotent (read-only; hardcoded in skeleton) | Default (3 retries, exponential backoff) |
| `ClassifyDocument` | Container name, blob name, correlationId | `ConfidenceResult<Classifications>` | Idempotent (hardcoded in skeleton) | Default |
| `PersistResult` | Storage account, container, blob name, content bytes, correlationId | `bool` (success) | Idempotent (blob upsert; no-op in skeleton) | Default |
| `ExtractInvoice` | Container name, blob name, page range, correlationId | `ConfidenceResult<Invoice>` | Idempotent (hardcoded in skeleton) | Default |
| `ValidateInvoice` | Name, `Invoice` data, correlationId | `ValidationResult` | Idempotent (pure logic) | Default |
| `ExtractContract` | Container name, blob name, page range, correlationId | `ConfidenceResult<ContractData>` | Idempotent (hardcoded in skeleton) | Default |
| `ValidateContract` | Name, `ContractData`, correlationId | `ValidationResult` | Idempotent (pure logic) | Default |
| `DetectBoundaries` | Container name, blob name, correlationId | `BoundaryDetectionResult` | Idempotent (hardcoded in skeleton) | Default |
| `ExtractGeneralDocument` | Container name, blob name, page range, correlationId | `ConfidenceResult<GeneralDocumentData>` | Idempotent (hardcoded in skeleton) | Default |

**Root Orchestrator Flow** (`ProcessDocumentBatchWorkflow`):
```
1. CallActivityAsync<DocumentFolders>(GetDocumentFolders, request)
2. Fan-out: foreach folder => CallSubOrchestrationAsync<WorkflowResult>(ProcessDocumentWorkflow, folder)
3. Fan-in: Task.WhenAll equivalent => collect all WorkflowResult
4. Aggregate all results => return root WorkflowResult
```

**Per-Folder Orchestrator Flow** (`ProcessDocumentWorkflow`):
```
foreach documentFileName in folder.DocumentFileNames:
  1. result = CallActivityAsync<ConfidenceResult<Classifications>>(ClassifyDocument, {folder, file})
  2. CallActivityAsync<bool>(PersistResult, {classification result as JSON})
  3. if result == null => log error, continue
  4. if result.OverallConfidence < ConfidenceThreshold => log error, add to errors, continue
  5. foreach pageClassification in result.Data.PageClassifications:
     - Invoice: ExtractInvoice => PersistResult => ValidateInvoice => PersistResult
     - Contract: ExtractContract => PersistResult => ValidateContract => PersistResult
     - BoundedDocument: DetectBoundaries => fan-out per segment => fan-in => PersistResult
     - General: ExtractGeneralDocument => PersistResult
     - Email/None: log, continue (no error entry)
return WorkflowResult (aggregated messages + errors)
```

**Per-Segment Orchestrator Flow** (`ProcessDocumentSegmentWorkflow`):
```
1. ClassifyDocument (for segment page range)
2. PersistResult (classification)
3. Route by classification => type-specific extraction + validation + persistence
   (BoundedDocument nested => fallback to General path)
4. return WorkflowResult
```

**State and Determinism Considerations**:
- Orchestrators must NOT call `DateTime.UtcNow`, `Guid.NewGuid()`, `Random`, or any I/O
- Use `context.CreateReplaySafeLogger()` for all logging within orchestrators
- Correlation ID is passed as part of the orchestration input (set by trigger); not read mid-replay
- All non-deterministic operations are in activities

**Error Handling and Compensation**:
- Transient activity failures: default retry policy (3 attempts, exponential backoff, first retry after 5 seconds)
- Classification failure or null result: log error, add error entry to `WorkflowResult`, continue to next document
- Confidence below threshold: log, add error entry, continue to next document
- Empty segment list from `DetectBoundaries`: add error entry for that document, continue (per FR-004)
- BoundedDocument nested classification within a segment: fallback to General path
- Queue message deserialization failure: dead-lettered after retry exhaustion; no orchestration started
- HTTP deserialization failure: HTTP 400 returned immediately; no orchestration started

## Security

**Identity and Access**:
- Local development: no managed identity required; Azurite uses development storage credentials
- Cloud (deferred): system-assigned Managed Identity with least-privilege RBAC roles per the spec

**Secret and Key Management**:
- No secrets required for local development
- Cloud: all connection strings and API keys in Azure Key Vault; referenced via App Settings Key Vault references; never in source control
- `local.settings.json` is git-ignored

**Network and Data Protection**:
- Local: no network isolation
- Cloud (deferred): TLS for all external communication; Azure-managed encryption at rest
- Skeleton phase: no real document data processed; no PII or financial data at risk

**Threat Considerations**:
- HTTP trigger is unauthenticated in local development; authentication deferred to cloud deployment phase
- `X-Correlation-Id` header value is used as a structured log property only; no injection risk

## Monitoring

**Telemetry**:

Every log entry MUST include as structured properties:
- `orchestrationInstanceId`
- `correlationId`

Key log events:

| Event | Level |
|---|---|
| HTTP/Queue trigger received | Information |
| Root orchestration started | Information |
| Folder discovery completed | Information |
| Sub-orchestration started per folder | Information |
| Document classified | Information |
| Document skipped (low confidence) | Warning |
| Document skipped (Email/None) | Information |
| Activity completed | Debug |
| Orchestration completed | Information |
| Orchestration failed | Error |

**Distributed Tracing**:
- `Microsoft.Azure.Functions.Worker.ApplicationInsights` emits automatic telemetry for Function invocations
- `orchestrationInstanceId` + `correlationId` in all log scopes enables cross-activity correlation in Application Insights

**Metrics**:
- Activity execution duration: captured automatically via Function invocation telemetry
- End-to-end orchestration duration: observable via orchestration start/complete timestamps
